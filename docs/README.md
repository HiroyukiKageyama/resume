#### 交通情報のビッグデータ解析
- 概要
    - 自動車の交通情報データを、顧客の提示した要求に従って解析する業務。
    - 数億~数十億という規模のデータを扱った。
    - この解析が渋滞予測の精度向上につながり、車社会の利便性に寄与できた。
- プロジェクト規模
    - 4人
- 担当した役割
    - 要件定義
        - 顧客の提示資料を細部まで読み込み、不明瞭な点をピックアップして質問し、ブラッシュアップして要件を明確化した。
        - 最初に要求をできる限り明確化したことで、仕様の認識齟齬による手戻りは発生しなかった。
    - インフラ
        - 大量のデータを処理するのに、どのくらいのプロセスを並行で処理する必要があるかを計算し、それに見合ったAWSのリソースを選定した。
    - バックエンド
        - データ整形のロジックを設計・実装。(Groovy)
        - 当初は数人で担当する予定だったが、仕様の理解度が高くミスが起こりにくいということで一人で実装を任された。
- 業務から得た経験・知見
    - 顧客の要望には注意深く考えるとより明確化できる余地が隠れていて、それを明確にすればするほど手戻りのリスクを抑えることができる。
    - 大量のデータ解析には並列処理が不可欠であり、リソース消費の少ない実装を心がけることが重要。
    - 数日規模という長時間の処理を、大幅な手戻りなく制御するための工夫。例えば、あらかじめ決めておいた件数ごとにバックアップを取り、リカバリできる手段を確保しておく等。
    

#### 自動車の税金に関する公共システムの開発
- 概要
    - 全国的な法律改正に伴う自動車の税金に関する公共システムの導入とそのカスタマイズ業務。
    - 自動車税、自動車取得税などに関する知識が必要。
    - 顧客の担当者がかなり高圧的な方で、前任者が退場している。
- プロジェクト規模
    - 3人
- 担当した役割
    - 要件定義
        - 公共システムの仕様書として配布される数百ページにも及ぶ資料を読み込み、仕様を理解する。
        - 資料を読むだけでは理解できない部分を、他のプロジェクトの詳しい人に話を聞きに行き理解する。
        - 業務担当者に公共システムの基本機能の説明を丁寧に行い、業務効率化のための要望を掬い上げて明確化していく。
    - バックエンド
        - 要望を仕様書に落とし込んで機能を実装する。(Java)
- 業務から得た経験・知見
    - 全く予備知識のない業務でも０から業務知識を身に着け、仕様を理解する能力。
    - 業務知識の重要性。技術力だけは仕事は成り立たず、むしろ業務知識のあるエンジニアは大変重宝される。
    - 高圧的な顧客に対する心構えや行動の工夫。

    - 16人
- 担当した役割

        - 保険会社側のSEから、基本設計と処理の概要が説明され、それを基に詳細設計書を作成し実装する。
        - 11人のエンジニアが同時に実装するにもかかわらず、汎用的なモジュール（DB接続用のクラス等）が一切無い状態だったので自発的に全体として必要なものを洗い出して実装し、提供した。

    - 大規模な開発現場においては自分だけではなく、全体の最適化を考慮し、自発的にリーダーシップを取って開発環境を整えることが重要。
